# ARCHITECTURE

## 1. 设计目标

ClawFounder 的目标不是构建一个泛化聊天助手，而是构建一套面向 `OPC（One-Person Company）` 的项目推进系统。

系统需要同时满足三件事：

1. 能够推进项目生命周期，而不是停留在单点任务
2. 能够把内容、上线、反馈纳入同一个循环
3. 能够通过现实接口层补足桌面端自动化的边界

---

## 2. 三层架构

### A. 编排层（Orchestration Layer）

负责项目状态推进。

核心职责：

- Idea Intake
- Spec Generation
- Prototype Handoff
- Launch Assets
- Growth Loop

这一层负责决定“下一步应该做什么”，而不是只回答“某一步怎么做”。

### B. 现实接口层（Real-world Interface Layer）

负责把移动端专属动作接入统一工作流。

核心定位：

- 通过 ADB 接管独立手机
- 将桌面端无法完整执行的移动端动作桥接进流程
- 作为系统与真实平台环境之间的设备接口层

这一层是 ClawFounder 区别于普通 Agent 展示的重要结构。

### C. 输出层（Output Layer）

负责产出可评审、可访问、可传播的结果。

主要输出：

- 白皮书 / 产品规格
- 原型 / GitHub 仓库
- 官网 / GitHub Pages
- 海报 / 主视觉 / 说明页
- 文章 / 推广素材 / 阶段汇报

---

## 3. 为什么需要现实接口层

普通 Agent 展示通常止步于：

- 查资料
- 写代码
- 改文案
- 生成页面

但真实项目推进中，经常会出现只能由移动端完成或确认的动作。

如果这些动作完全脱离主工作流，系统会在关键节点断开。

因此 ClawFounder 采用 `ADB + 独立手机` 的方式，把这类动作抽象为“移动端动作桥接”，而不是把它们视为工作流之外的人工步骤。

---

## 4. 与普通演示型 Agent 的区别

普通演示：

- 强调单点能力
- 任务多但缺少生命周期视角
- 很难解释项目如何持续推进

ClawFounder：

- 强调项目推进链
- 强调产物的可访问、可传播、可复盘
- 强调现实接口层的结构性作用

---

## 5. 当前仓库中的架构呈现方式

本仓库用三种方式呈现架构：

1. `index.html` 中的评审型页面与架构图
2. `poster.html` / `poster-v2.png` 中的宣传海报视觉
3. `docs/` 中的文字化架构说明

后续如果进入正式工程阶段，可以把当前概念结构拆为真正的模块仓库或服务边界。
